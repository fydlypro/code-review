## CRITIQUE (action immédiate)

### [C-1] RLS : SELECT merchants et customers publiquement accessible

Fichier : fydly_fix_rls.sql (appliqué en production)

La policy suivante contient OR true, ce qui annule toute restriction :

  USING (auth.uid() = user_id OR true)

Impact : N'importe qui (même non authentifié via la clé anon) peut lire
TOUTES les données de TOUS les commerçants et clients (nom, email, téléphone,
statut abonnement, Stripe ID, GDPR timestamps).

Correction :
  DROP POLICY IF EXISTS "merchants_select_own_v2" ON public.merchants;
  DROP POLICY IF EXISTS "customers_select_own_v2" ON public.customers;

  CREATE POLICY "merchants_select_own_v3"
    ON public.merchants FOR SELECT
    USING (auth.uid() = user_id);

  CREATE POLICY "customers_select_own_v3"
    ON public.customers FOR SELECT
    USING (auth.uid() = user_id);

---

### [C-2] RLS : INSERT non authentifié autorisé sur merchants et customers

Fichier : fydly_fix_rls.sql

  WITH CHECK (true);  -- n'importe qui peut insérer

Impact : Un utilisateur non authentifié peut créer des enregistrements
arbitraires dans merchants et customers — pollution de base, attaques
par collision, comptes frauduleux.

Correction :
  DROP POLICY IF EXISTS "merchants_insert_own_v2" ON public.merchants;
  DROP POLICY IF EXISTS "customers_insert_own_v2" ON public.customers;

  CREATE POLICY "merchants_insert_own_v3"
    ON public.merchants FOR INSERT
    WITH CHECK (auth.uid() = user_id AND auth.uid() IS NOT NULL);

  CREATE POLICY "customers_insert_own_v3"
    ON public.customers FOR INSERT
    WITH CHECK (auth.uid() = user_id AND auth.uid() IS NOT NULL);

### [C-3] Edge Functions cron : auth bypass si CRON_SECRET non défini

Fichiers :
  supabase/functions/rotate-qr-tokens/index.ts
  supabase/functions/expire-rewards/index.ts

  const cronSecret = Deno.env.get("CRON_SECRET");
  if (cronSecret && authHeader !== `Bearer ${cronSecret}`) {
    // Ce bloc N'EST PAS EXÉCUTÉ si CRON_SECRET est absent
    return 401;
  }

Impact : Si CRON_SECRET n'est pas défini dans les secrets Supabase, toute
requête non authentifiée peut forcer la rotation de tous les QR codes ou
expirer toutes les récompenses.

Correction (remplacer le && par la vérification inversée) :
  if (!cronSecret || authHeader !== `Bearer ${cronSecret}`) {
    return new Response(JSON.stringify({ error: "unauthorized" }), { status: 401 });
  }

Vérifier aussi que CRON_SECRET est bien défini :
  Supabase Dashboard → Settings → Edge Functions → Secrets

## HAUTE SÉVÉRITÉ

### [H-1] RLS : INSERT transactions sans restriction de scope

Fichier : fix_all_policies.sql

  CREATE POLICY "transactions_insert_authenticated"
    ON public.transactions FOR INSERT
    TO authenticated
    WITH CHECK (true);

Impact : Tout utilisateur authentifié peut insérer une transaction earn
avec n'importe quel merchant_id / customer_id. Auto-attribution de tampons
sans scanner de QR.

Correction : Supprimer cette policy — l'insertion doit passer uniquement
par la fonction upsert_stamp (SECURITY DEFINER) :
  DROP POLICY IF EXISTS "transactions_insert_authenticated" ON public.transactions;

---

### [H-2] CORS : wildcard * sur toutes les Edge Functions sensibles

Toutes les fonctions utilisent : "Access-Control-Allow-Origin": "*"
Y compris delete-customer-account, delete-merchant-account, create-checkout.

Impact : N'importe quel site web peut déclencher des requêtes cross-origin
sur ces endpoints.

Correction :
  "Access-Control-Allow-Origin": "https://votre-domaine-production.com"

---

### [H-3] Open Redirect : success_url / cancel_url / return_url non validés

Fichiers :
  supabase/functions/create-checkout/index.ts
  supabase/functions/create-portal/index.ts

Les URLs viennent du body client et sont passées directement à Stripe
sans validation de domaine. Redirection vers n'importe quel domaine
externe possible après paiement.

Correction : Valider les URLs contre une allowlist avant de les passer à Stripe :

  const ALLOWED_ORIGINS = ["https://votre-domaine.com"];
  function validateRedirectUrl(url: string): boolean {
    try {
      return ALLOWED_ORIGINS.includes(new URL(url).origin);
    } catch { return false; }
  }

### [H-4] verify_jwt non configuré pour les Edge Functions

Constat : Aucun fichier config.toml dans aucun dossier de fonction.
Les commentaires indiquent "verify_jwt: true" mais SANS config.toml
c'est le comportement par défaut Supabase qui s'applique.

Impact : Les fonctions qui décodent manuellement le JWT (getUserIdFromJwt)
sans vérifier la signature pourraient accepter des JWTs forgés si
verify_jwt n'est pas actif au niveau gateway.

Correction : Créer config.toml dans chaque fonction nécessitant l'auth :

  # supabase/functions/send-push-notification/config.toml
  [functions.send-push-notification]
  verify_jwt = true

## MOYEN

### [M-1] Rate limiting login uniquement côté client (localStorage)

Le rate limiting de connexion repose sur localStorage, triviallement
contournable (clear storage, mode privé, autre navigateur).

Recommandation : Activer le rate limiting natif Supabase Auth :
  Supabase Dashboard → Authentication → Rate Limits

---

### [M-2] QR tokens lisibles par tous (anon)

  CREATE POLICY "qr_tokens_select_public" ON public.qr_tokens FOR SELECT USING (true);

Tous les tokens QR actifs de tous les commerçants sont accessibles
avec la clé anon.

Recommandation : Restreindre l'accès direct et passer par le RPC
validate_qr_token uniquement :
  DROP POLICY IF EXISTS "qr_tokens_select_public" ON public.qr_tokens;
  CREATE POLICY "qr_tokens_merchant_only"
    ON public.qr_tokens FOR SELECT
    USING (
      EXISTS (SELECT 1 FROM merchants m
              WHERE m.id = qr_tokens.merchant_id
                AND m.user_id = auth.uid())
    );

---

### [M-3] Messages d'erreur internes exposés en 500

Toutes les Edge Functions retournent error.message en réponse 500,
potentiellement des stack traces Deno/Stripe.

Recommandation :
  console.error("[func] Erreur:", error);
  return new Response(JSON.stringify({ error: "Erreur interne." }), { status: 500 });

---

### [M-4] Dépendances Deno std très anciennes

Toutes les fonctions importent deno.land/std@0.177.0 (2023).
Version actuelle : ~0.224.x.

Recommandation : Mettre à jour vers la version stable courante.

## LOW

### [L-1] isAdmin vérifié seulement via app_metadata côté client

La vérification est correcte (app_metadata non modifiable par l'user),
mais si le refresh de session échoue silencieusement, l'état peut être périmé.

Recommandation : Vérifier le rôle admin côté Edge Function pour les
actions admin critiques.

---

### [L-2] HTTPS non forcé par en-tête HSTS

enforceHTTPS() est client-side uniquement.

Recommandation : Ajouter Strict-Transport-Security dans vercel.json :
  "headers": [{ "key": "Strict-Transport-Security", "value": "max-age=31536000; includeSubDomains" }]

---

### [L-3] Pas de Content Security Policy (CSP)

Aucun en-tête CSP détecté dans vercel.json.

Recommandation minimale :
  Content-Security-Policy: default-src 'self'; connect-src 'self' https://*.supabase.co https://onesignal.com
